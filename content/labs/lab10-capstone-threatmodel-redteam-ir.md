이 실습은 12주(3개월) AI 보안 캡스톤 프로젝트의 마지막 단계입니다. [Lab 9: 캡스톤 2개월차 — 보안 통제 설계와 공급망 보안 점검](../lab9-capstone-controls-supplychain/)에서 개선한 v2를 대상으로, MITRE ATLAS 기반 위협모델을 작성하고, 정식 레드팀 라운드를 수행합니다. 이어서 [Lab 3: AI 위험 등록부 작성하기](../lab3-ai-risk-register/)의 형식을 NIST GenAI Profile 관점으로 확장한 리스크 등록부와 사고대응 플레이북을 작성합니다.

이 단계를 마치면 12주 캡스톤의 최종 4종 산출물 — ① 취약한 AI 앱(Lab 8), ② 개선된 AI 앱(Lab 9), ③ AI 레드팀 리포트(이 Lab의 10주차), ④ AI 보안 설계 문서(이 Lab의 9·11·12주차) — 가 모두 완성됩니다.

## 목표

- 시스템을 Environment / AI Platform / AI Model / AI Data 네 영역으로 나누고, MITRE ATLAS 전술을 최소 8개 시나리오에 매핑한다.
- 정식 레드팀 라운드(수동 테스트 20건 이상, 주요 발견사항 5건 이상)를 계획·실행하고 로그를 남긴다.
- NIST GenAI Profile의 위험 범주를 반영한 리스크 등록부(위험 10개 이상)와 운영 모니터링 체크리스트를 작성한다.
- 사고 대응 플레이북(탐지→차단→조사→복구→보고)을 작성하고, 12주 캡스톤 최종 보고서를 완성한다.

{{< callout type="info" >}}
가능하다면 동료 1명을 참여시켜 "일반 사용자 관점"과 "공격자 관점"으로 역할을 나누어 같은 시스템을 테스트해 보세요. Microsoft AI Red Team 가이드에서도 강조하듯, 같은 시스템이라도 보는 관점에 따라 발견되는 항목이 크게 달라집니다. 혼자 진행한다면, 테스트 라운드를 두 번으로 나누어 "이 라운드는 일반 사용자처럼", "이 라운드는 공격자처럼"이라는 의식적인 페르소나 전환을 적용하는 것만으로도 효과를 볼 수 있습니다.

#### 시스템 요소 분류표

[MITRE ATLAS](../../docs/governance/mitre-atlas/)의 4-레이어 모델을 그대로 사용해, 이 RAG 챗봇 프로젝트의 실제 구성요소를 채웁니다.

| 시스템 요소 | 정의 | 이 RAG 챗봇 프로젝트에서의 예시 |
|---|---|---|
| Environment | 시스템이 운영되는 외부 환경 및 사용자 접점 | 호스팅 서버(클라우드 VM/컨테이너), 네트워크 경계, API 게이트웨이, 사용자 인증 체계 |
| AI Platform | 모델을 서빙/운영하는 인프라 및 오케스트레이션 계층 | API 게이트웨이, 벡터DB(임베딩 검색), 프롬프트 오케스트레이션 코드, 출력 후처리/필터링 모듈 |
| AI Model | 모델 자체 (가중치, 아키텍처, 추론 로직) | 외부 LLM API(채팅 완성 모델), 임베딩 모델 |
| AI Data | 학습/운영에 사용되는 데이터 | RAG 문서 코퍼스(사내 위키), 벡터 인덱스(임베딩), 대화 로그, 시스템 프롬프트 |

| Initial Access | 인증 없이 공개 API 엔드포인트에 접근해 챗봇과 직접 상호작용 | Environment | Lab 8: 인증 없는 엔드포인트에서 무제한 질의 가능 | Lab 9: API 키/레이트리밋 적용, 엔드포인트 접근 제어 |

위협모델 다이어그램은 mermaid로 작성해도 좋습니다. 4개 시스템 요소를 박스로, 위 표의 8개 시나리오를 화살표로 표시하면 [MITRE ATLAS](../../docs/governance/mitre-atlas/) 문서의 4-레이어 다이어그램과 동일한 형식이 됩니다.

### 10주차: 정식 레드팀 라운드

| 항목 | 내용 |

| 해야 할 일 | 레드팀 계획 작성 → 20건 이상 테스트 수행 → 레드팀 로그 작성 → 주요 발견사항 5건 이상 정리 |
| 산출물 | `redteam-plan.md`(계획), `redteam-log.md`(테스트 로그), 주요 발견사항 요약 |

#### 레드팀 계획 템플릿

- **범위(Scope)**: 테스트 대상 시스템(v2 엔드포인트), 테스트 제외 대상(프로덕션 데이터, 실제 사용자 계정 등)을 명시
- **참여자/역할**: 혼자 진행할 경우에도 "일반 사용자 관점"과 "공격자 관점"을 라운드별로 구분해 기록
- **테스트 규칙(Rules of Engagement)**: 테스트 시간대, 허용되는 요청 빈도(DoS 유발 금지), 실제 운영 환경이 아닌 로컬/스테이징 환경 사용 여부
- **기록 양식**: 모든 시도를 테스트ID, 입력, 응답, 판정으로 기록하고 증거(스크린샷/로그 파일 경로)를 남길 위치를 사전에 정의

| RT-002 | 2026-06-01 | 악성 문서를 RAG 코퍼스에 업로드 후 간접 인젝션 시도 | Persistence / LLM01 | 일부 지시문이 응답에 반영됨 | 부분 실패 | 높음 | `logs/rt-002.txt` |

### 11주차: NIST식 리스크 등록부와 운영 통제

| 해야 할 일 | 확장 리스크 등록부 템플릿 채우기(위험 10개 이상) → 각 위험을 GenAI Profile 범주로 분류 → 운영 모니터링 체크리스트 작성 |

#### 확장 리스크 등록부 템플릿

[Lab 3](../lab3-ai-risk-register/)의 위험 등록부 컬럼(기능/목적, 입력 데이터/툴 권한, 위협 시나리오, 위험 등급, 영향, 발생 가능성, 현재 통제, 잔여 위험, 모니터링 지표, 담당자, 중지 조건)에 NIST GenAI Profile 12개 위험 범주를 분류 축으로 추가한 형태입니다.

NIST GenAI Profile 12개 위험 범주: confabulation(환각), CBRN 정보 보조, 위험한/폭력적/유해 콘텐츠, 데이터 프라이버시, 환경적 영향, 인간-AI 형상화(anthropomorphization), 정보 무결성, 정보보안, 지적재산, 유해한 편향과 동질화, 부적절한 사용 사례, 가치사슬·구성요소 통합

| 위험 ID | GenAI Profile 범주 | 기능/목적 | 위협 시나리오(ATLAS) | 위험 등급(EU AI Act) | 영향 | 발생 가능성 | 현재 통제 | 잔여 위험 | 모니터링 지표 | 담당자 | 중지 조건 |
|---|---|---|---|---|---|---|---|---|---|---|---|
| RISK-01 | 정보보안 | RAG 문서 검색 응답 생성 | Execution — 간접 프롬프트 인젝션(Lab 9 RT-002) | 제한적 위험 | 잘못된 지시 실행, 응답 변조 | 중 | 입력/문서 구분 태그, 시스템 프롬프트 강화 | 신규 인젝션 패턴에 취약 가능 | 인젝션 탐지 룰 트리거 횟수(일별) | 보안팀 | 인젝션 성공률 임계치 초과 시 RAG 검색 일시 중단 |
| RISK-02 | 데이터 프라이버시 | 코퍼스 내 더미 개인정보 응답 노출 | Exfiltration — PII 회수(Lab 8 시나리오 C) | 고위험(개인정보 포함 시) | 개인정보 노출, 법적 리스크 | 낮음 | 응답 PII 마스킹 필터 | 마스킹 패턴 우회 가능성 | PII 패턴 매칭 차단 건수 | 개발팀 | PII 노출 1건이라도 확인 시 즉시 비활성화 |
| RISK-03 | confabulation(환각) | 정책/매뉴얼 질의에 대한 부정확한 답변 생성 | Impact — 잘못된 정보 기반 의사결정 유도 | 제한적 위험 | 고객 불만, 잘못된 의사결정 | 중 | 출처 문서 인용 강제(citation 표시) | 인용은 있지만 해석 오류 가능 | 사용자 피드백("도움 안 됨") 비율 | 콘텐츠팀 | 환각 응답 신고율 임계치 초과 시 경고 배너 노출 |
| RISK-04 | 가치사슬·구성요소 통합 | 외부 LLM API/임베딩 모델/벡터DB 조합 운영 | Resource Development — 공급망 구성요소 취약점 | 제한적 위험 | 의존 컴포넌트 취약점으로 인한 전체 시스템 영향 | 중 | pip-audit, SBOM 생성(Lab 7) | 신규 CVE 발생 시 대응 지연 가능 | SBOM 재생성/스캔 주기 준수 여부 | 인프라팀 | 핵심 의존성 Critical CVE 발견 시 즉시 패치/롤백 |

{{< callout type="info" >}}
실제 작성 시에는 위 4개 예시 외에 최소 6개 위험을 추가해 10개 이상을 채워야 합니다. 9주차 ATLAS 매핑 표의 8개 시나리오, 10주차 레드팀 로그의 주요 발견사항을 각각 위험 ID로 변환하면 자연스럽게 10개를 넘기게 됩니다.

### 12주차: 사고대응 플레이북과 최종 발표

| 항목 | 내용 |
|---|---|
| 목표 | 사고대응 플레이북을 작성하고, 12주 캡스톤 전체 산출물을 하나의 최종 보고서로 통합한다 |
| 해야 할 일 | 5단계 사고대응 플레이북 작성 → 최종 보고서 구성(4종 산출물 통합) → 자가 평가 수행 |
| 산출물 | `incident-response-playbook.md`, 12주 캡스톤 최종 보고서 |
| 통과 기준 | 플레이북이 탐지→격리→조사→복구→보고 5단계 모두 구체적인 트리거/조치를 포함하고, 최종 보고서가 4종 산출물(취약한 앱/개선된 앱/레드팀 리포트/보안 설계 문서)을 모두 참조한다 |

#### 사고대응 플레이북 템플릿

| 단계 | 이 RAG 챗봇에서의 트리거 조건 | 수행할 조치 |
|---|---|---|
| 탐지(Detection) | 비정상적으로 많은 PII 포함 응답 로그 발생, 또는 동일 인젝션 패턴이 단시간에 반복 탐지됨 | 모니터링 대시보드 알림 발생, 보안팀에 자동 알림(Slack/이메일) 전송 |
| 격리/차단(Containment) | 알림 발생 후 보안팀이 영향 범위를 1차 확인 | 해당 API 키 비활성화 또는 RAG 검색 엔드포인트 일시 중단, 필요 시 전체 챗봇 서비스 중단 |
| 조사(Investigation) | 격리 완료 후 | 관련 로그(요청/응답 전체), 입력 프롬프트, 인덱스 변경 이력을 시간순으로 추적해 원인(인젝션 페이로드, 악성 문서, 설정 오류 등) 식별 |
| 복구(Recovery) | 원인 식별 완료 | 영향받은 문서를 RAG 인덱스에서 제거하거나 이전 버전으로 롤백, 시스템 프롬프트/필터 설정을 안전한 버전으로 복원, 재발 방지 통제 추가 후 서비스 재개 |
| 보고/커뮤니케이션(Reporting) | 서비스 재개 후 | 영향 범위(노출된 데이터 유형, 영향받은 사용자 수 추정), 원인, 조치 내역, 재발 방지 계획을 정리해 관련자에게 공유. 리스크 등록부(11주차)의 해당 위험 항목 업데이트 |

#### 최종 보고서 구성 안내

12주 캡스톤의 최종 산출물은 4종입니다. 최종 보고서는 이 4종을 하나의 문서(또는 위키/포트폴리오 페이지)로 묶는 작업입니다.

1. **취약한 AI 앱(Lab 8)**: v0 코드 저장소 링크 + 적용한 공격 시나리오 A~D 결과
2. **개선된 AI 앱(Lab 9)**: v2 코드 저장소 링크 + 적용한 보안 통제와 v0/v2 비교
3. **AI 레드팀 리포트(10주차)**: 레드팀 계획, 20건 이상의 테스트 로그, 주요 발견사항 5건 이상을 [포트폴리오 프로젝트](../../docs/red-teaming/portfolio-projects/)에서 설명한 "요약 → 본문 → 부록" 구조로 정리
4. **AI 보안 설계 문서(9·11·12주차 통합)**: 9주차 위협모델(시스템 요소 분류 + ATLAS 매핑) + 11주차 확장 리스크 등록부 및 운영 모니터링 체크리스트 + 12주차 사고대응 플레이북을 하나의 문서로 합친 것

네 산출물은 서로를 참조해야 합니다. 예를 들어 보안 설계 문서의 리스크 등록부 항목은 레드팀 리포트의 발견사항을 인용하고, 사고대응 플레이북의 트리거 조건은 운영 모니터링 체크리스트의 점검 항목과 일치해야 합니다.

## 체크리스트

- [ ] 9주차: 시스템을 Environment/AI Platform/AI Model/AI Data 4개 요소로 분류하고 구체적 구성요소를 채웠다.

- [ ] 10주차: 레드팀 계획(범위, 참여자/역할, 테스트 규칙, 기록 양식)을 작성했다.
- [ ] 10주차: 테스트 20건 이상, 주요 발견사항 5건 이상을 로그로 기록했다.
- [ ] 11주차: NIST GenAI Profile 12개 범주를 결합한 확장 리스크 등록부에 위험 10개 이상을 등록했다.
- [ ] 11주차: 운영 모니터링 체크리스트(점검 항목/주기/담당)를 작성했다.

- [ ] 12주차: 4종 최종 산출물(취약한 앱/개선된 앱/레드팀 리포트/보안 설계 문서)을 하나의 최종 보고서로 통합했다.

## 학습효과

- **위협모델링의 실전 적용**: MITRE ATLAS의 4-레이어 모델을 추상적인 이론이 아니라, 직접 만든 RAG 챗봇의 구성요소와 공격 시나리오에 매핑해봄으로써 "프레임워크를 안다"와 "프레임워크를 적용할 수 있다"의 차이를 체감합니다.
- **정식 레드팀 프로세스 경험**: 즉흥적인 시도가 아니라 계획-실행-로그-발견사항이라는 정식 프로세스로 20건 이상의 테스트를 수행한 경험은, 면접에서 "레드팀을 해봤다"는 주장에 구체적인 근거를 제공합니다.
- **3개월 캡스톤의 완성**: 1개월차의 취약한 앱 구현부터 2개월차의 보안 통제 설계, 3개월차의 위협모델·레드팀·리스크 등록부·사고대응 플레이북까지 — LLM 앱을 직접 공격하고, 방어하고, 문서화해본 이 경험 전체가 하나의 포트폴리오 스토리가 되어, AI 보안 관련 직무에서 가장 강력한 차별점이 됩니다.
